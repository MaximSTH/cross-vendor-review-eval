function _parse_rfc2231(params, str) {
  // Fast-path: an RFC 2231 parameter always contains both `*` and `=`; skip the
  // scan for the common header (Subject, From, ...) that has neither.
  if (str.indexOf('*') === -1 || str.indexOf('=') === -1) return

  for (const { attribute, section, value: raw } of tokenize(str)) {
    let value = raw

    const lang_match = /^(.*?)'(.*?)'(.*)/.exec(value)
    if (lang_match) {
      if (attribute === params.cur_key && lang_match[2] !== params.cur_lang) {
        continue
      }
      params.cur_enc = lang_match[1]
      params.cur_lang = lang_match[2]
      value = lang_match[3]
    } else if (attribute !== params.cur_key) {
      params.cur_lang = ''
      params.cur_enc = ''
    }

    params.cur_key = attribute
    params.keys[attribute] = ''
    params.kv[`${attribute}*${section}`] = value
  }
}

I reviewed the diff against the original logic and the `haraka-message-stream` / `haraka-email-message` internals it depends on.

The change is well-motivated — the read path in `haraka-message-stream` re-emits headers from `header_list` and then skips stored bytes until it finds a blank line (`process_buf`, `headers_found_eoh`), so a stored message with no separator silently loses its entire body. Synthesizing a separator fixes that, and moving the `add_header`/`remove_header` guards from `header_pos > 0` to `found_hb_sep` is correct (`header_pos` was 0 for a message whose DATA opened with the separator).

The problem is where the busted-email heuristic now runs.

**Eager header-block termination.** The old code applied the "first line that doesn't look like a header" scan only in `end_data`, and only when `!this.found_hb_sep` — i.e. exclusively for messages that never had a blank line at all. The new code applies `header_re` to *every* line during streaming, so the first non-matching line terminates the header block even when a proper separator arrives later.

`Header.parse` (node_modules/haraka-email-message/lib/header.js:100-107) tolerates junk lines — it logs `Header did not look right` and moves on. So previously a header block containing one malformed line still parsed every other header. Now everything from that line onward is never handed to `header.parse` at all; it becomes body text, plus a spurious blank line is injected into the stored message.

Realistic triggers: a UTF-8 BOM before the first header (`\uFEFF` is in JS `\s`, so `[^\s:]*` can't match it and the alternation fails at position 0), an mbox-style `From user@host Mon Jan 1 ...` line, or any `X Bad: value`. A message like `BOM+Received:` / `Subject:` / `From:` / blank / body loses `Subject` and `From` entirely.

Note the new tests can't catch this — every busted case they cover has no separator anywhere, which is exactly the case the old code already handled.
